# 知识编译 vs 知识检索（Compiled vs Retrieved Knowledge）

## 概述

这是 LLM 知识管理中的一个核心架构选择：知识应该在收录时编译（compile），还是在查询时检索（retrieve）？

编译式知识在来源被添加时就进行一次性处理 —— 提取实体、更新摘要、建立交叉引用、标记矛盾。查询时直接读取已编译的结果。

检索式知识（即 RAG 模式）将原始文档切块并嵌入向量空间，在每次查询时重新搜索、拼凑、综合。

这与编程语言中"编译器 vs 解释器"的权衡完全类似。

## 核心思想

- **编译式**（LLM 维基）：收录时付出成本（LLM 调用），查询时几乎免费（读取 Markdown）
- **检索式**（RAG）：收录时成本低（切块 + 嵌入），查询时成本高（搜索 + 重排 + LLM 综合）
- 编译式在跨文档综合方面天然更强 —— 交叉引用已预先建立
- 检索式在精确引用溯源方面更强 —— 可以指向具体的文本块
- 混合方案可能是最优解：编译层提供综合理解，检索层提供引用根据
- **这个模式不限于知识** —— AutoHarness 将同样的"编译 vs 运行时推理"逻辑应用于行为：LLM 将行为编译为代码（训练时），运行时直接执行代码而非调用 LLM [arXiv:2603.03329]

## 待解问题

- 编译式知识在来源数量增长到数千时，维护成本如何变化？
- 编译过程中的 LLM 错误（幻觉、遗漏）如何检测和纠正？
- 能否自动化检测"需要重新编译"的触发条件？
