# AutoHarness：通过自动合成代码安全线束改进 LLM 智能体（AutoHarness: Improving LLM Agents by Automatically Synthesizing a Code Harness）

## 核心要点

- LLM 智能体频繁执行非法动作 —— 在国际象棋比赛中，Gemini-2.5-Flash 78% 的败局归因于非法棋步，而非策略失误 [arXiv:2603.03329]
- AutoHarness 让 LLM 自动合成约束自身行为的代码安全线束（Harness），通过程序空间上的树搜索 + 汤普森采样（Thompson Sampling）迭代改进 [arXiv:2603.03329]
- 三种线束模式：动作过滤器（Action-Filter）、动作验证器（Action-Verifier）、纯代码策略（Policy）[arXiv:2603.03329]
- 在 145 个 TextArena 游戏中实现 100% 合法动作成功率 [arXiv:2603.03329]
- Gemini-2.5-Flash + Harness（56.3% 胜率）击败了更大的 Gemini-2.5-Pro（38.2%）[arXiv:2603.03329]
- 纯代码策略模式：平均奖励 0.870 > GPT-5.2-High 的 0.844，测试时成本接近零（vs GPT 实验的 $640）[arXiv:2603.03329]

## 摘要

本文提出了 AutoHarness，一种让 LLM 自动生成代码安全线束的方法。核心问题是：即使是先进的 LLM 智能体也频繁产生环境不允许的非法动作。与其让模型在权重层面内化所有规则，不如让模型自动合成一个可执行的代码约束层。

方法的核心是程序空间上的搜索：系统维护多个代码候选方案的树结构，使用汤普森采样（Thompson Sampling）选择下一个改进的候选方案，LLM 充当变异算子（Mutation Operator）—— 接收环境反馈（非法动作错误信息），输出改进后的代码。每次改进最多使用 5 个失败步骤的反馈，由"批评者"（Critic）整合错误模式后交给"改进者"（Refiner）生成新代码。平均仅需 14.5 次迭代即可收敛。[arXiv:2603.03329]

三种线束模式代表递进的自动化程度：动作过滤器生成合法动作集供 LLM 选择；动作验证器检验 LLM 提议的动作并拒绝非法操作；纯代码策略完全替代 LLM，测试时只运行 Python 代码。[arXiv:2603.03329]

## 值得关注的主张

- "将脆弱的智能体行为从原始的下一个 token 推理移入可执行代码" —— 这是一种将学习从神经网络权重转移到可检查代码的系统思想 [arXiv:2603.03329]
- 小模型 + 自动约束 > 大模型无约束 —— 这挑战了"用更大模型解决一切"的默认策略 [arXiv:2603.03329]
- 纯代码策略在不调用 LLM 的情况下超越了 GPT-5.2-High —— LLM 的知识可以被蒸馏为可执行代码 [arXiv:2603.03329]
- 作者提出未来将领域专家蒸馏回基础 LLM，形成递归自我改进循环 [arXiv:2603.03329]

## 局限性

- 每个游戏需要独立生成线束，无跨游戏迁移学习 [arXiv:2603.03329]
- 双人博弈的纯代码策略尚未解决 —— 需要对手建模（类 MCTS 方法）[arXiv:2603.03329]
- 论文未提供消融实验（Ablation Study）—— 无法确定树搜索、汤普森采样、批评者/改进者各自的贡献 [arXiv:2603.03329]
- 未讨论失败模式分析 [arXiv:2603.03329]

## Eric 的思考过程

收录本文时经历了几轮认知修正：

**最初的过度简化：** 一开始把 AutoHarness 理解为"带环境反馈的编程智能体" —— 类似 Cursor/Claude Code + 测试套件。这个类比抹去了论文的实际贡献。正确的理解是：它不是在写解决任务的代码，而是在合成约束自身行为的代码安全线束。LLM 是在构建自己的护栏，不是在构建解决方案。

**关于 RL 的辨析：** 这个方法有明显的 RL 味道（环境、rollout、反馈、探索-利用权衡），但不是标准 RL。标准 RL 优化的是神经网络权重（通过梯度下降、策略梯度等）；AutoHarness 优化的是代码（通过 LLM 重写 + 树搜索）。更准确的定位是"程序空间上的黑盒策略改进"或"LLM 引导的程序搜索"。最接近的 RL 子族是进化策略（Evolution Strategies）和蒙特卡洛程序搜索。

**关键区分 —— 权重中学习 vs 代码中学习：** RL 在权重中学习策略（不透明、不可检查）。AutoHarness 在代码中学习策略（透明、可检查、可执行）。这是一个重要的系统思想：当失败模式是规则遵守、合法性验证、接口正确性时，搜索可执行控制逻辑比试图让模型在权重中内化一切更高效、更可靠。

**与 LLM 维基模式的同构：** AutoHarness 和 LLM 维基是同一个元模式的不同实例 —— "LLM 作为编译器，而非解释器"。LLM 维基将知识编译为 Markdown 文件；AutoHarness 将行为编译为 Python 代码。两者都在收录/训练阶段做昂贵的 LLM 工作，在查询/运行阶段享受廉价的产物使用。

**可扩展性的问题：** 这个方法对封闭、确定性、完全可观测、快速反馈的环境效果很好（游戏天然具备这些性质）。对于开放、随机、部分可观测、延迟反馈的真实世界系统，扩展策略是分解（Decomposition）—— 将复杂系统拆解为足够"像游戏"的子任务，对这些子任务应用线束合成，其余部分保留 LLM 推理。
