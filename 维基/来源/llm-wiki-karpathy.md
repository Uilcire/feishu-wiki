# LLM 维基（LLM Wiki）—— 卡帕西

## 核心要点

- 传统 RAG 系统在每次查询时从零开始重新推导知识，没有积累效应
- LLM 维基模式让 LLM 在收录来源时一次性编译知识，构建并维护一个持久的、互相链接的 Markdown 维基
- 三层架构：原始资料（不可变来源）→ 维基（LLM 生成维护）→ 架构文档（约定和工作流）
- 三大操作：收录（Ingest）、查询（Query）、审查（Lint）
- 关键洞察：人类放弃维基是因为维护负担增长快于价值回报；LLM 让维护成本接近零
- 查询结果可以回写为维基页面，让探索过程也能持续积累
- 灵感来源于万尼瓦尔·布什（Vannevar Bush）1945 年提出的 Memex 概念

## 摘要

卡帕西提出了一种用 LLM 构建个人知识库的模式。核心思路是：不要像传统 RAG 那样在查询时才从原始文档中检索和拼凑答案，而是让 LLM 在收录新来源时就将知识"编译"进一个持久维基中 —— 更新实体页面、修订主题摘要、标记矛盾之处、加强或挑战已有的综合分析。

这个模式的架构分三层。最底层是不可变的原始资料（文章、论文、数据文件）。中间层是 LLM 完全拥有的维基 —— 一组互相链接的 Markdown 文件，包括摘要、实体页面、概念页面和综合分析。最上层是架构文档（如 CLAUDE.md），定义了维基的结构约定和工作流程，让 LLM 成为一个有纪律的维基维护者而不是通用聊天机器人。

在实际使用中，卡帕西描述了 Obsidian + LLM 智能体并排使用的工作方式："Obsidian 是 IDE；LLM 是程序员；维基是代码库。"他强调维基之所以能持续维护，是因为 LLM 不会厌倦、不会忘记更新交叉引用、能在一次操作中修改 15 个文件。人类的工作是整理来源、引导分析、提出好问题；LLM 负责其余一切。

## 值得关注的主张

- "知识应该在收录时编译一次，而不是在每次查询时重新推导" —— 这是对整个 RAG 范式的根本性批评
- "人类放弃维基是因为维护负担增长快于价值" —— 维基失败的根本原因不是工具或意愿问题，而是经济学问题
- "在中等规模（~100 个来源、数百个页面）下，索引文件 + LLM 读取的方式出奇地好用，无需嵌入式 RAG 基础设施" —— 对小到中规模知识库而言，简单方案可能优于复杂方案
- "好的查询答案可以回写为维基页面，让探索过程也能积累" —— 这模糊了"检索"和"创造"之间的边界
- 该模式是"有意抽象的" —— 只传达模式本身，具体实现留给用户和 LLM 协作完成

## Eric 的思考过程

收录本文时产生了几个关键洞察：

**1. LLM 维基本质上是低成本的知识图谱**

维基页面 = 图节点，wikilinks = 图边。与正式的图数据库（如 Neo4j）相比，LLM 维基用松散的自然语言链接代替了结构化的类型关系。优势是零基础设施、人类可直接阅读；劣势是无法做精确的多跳结构化查询。随着规模增长，这个松散图的可搜索性会退化。

**2. 可搜索性是这个架构的最大弱点**

卡帕西自己承认索引文件方案只在 ~100 来源规模下好用。没有语义搜索意味着无法做模糊匹配（"意思接近"的查找）。他提到 qmd 作为补救措施 —— 但这本质上就是在维基之上加了一层 RAG，等于承认纯维基模式不够。

**3. 维基页面本身应该被 RAG 索引（关键洞察）**

维基页面是原始文档的"再生产"，信息密度更高、结构更清晰。将它们也切块嵌入 RAG 索引，等于获得了两层检索：原始 chunks 提供精确引用，维基 chunks 提供综合理解。这是一种"通过预综合实现查询扩展"（query expansion by pre-synthesis）的策略。引用链变成：**答案 → 维基 chunk → 维基页面 → 原始 chunk → 原始文档**，两层溯源。

**4. 最优架构可能是三层检索 + 编译层**

向量搜索（语义）+ 词法搜索（关键词）+ 图搜索（关系遍历），再加上编译层（维基页面）同时参与所有三层检索。这样既有 RAG 的全面覆盖和精确引用，又有编译知识的综合深度。
